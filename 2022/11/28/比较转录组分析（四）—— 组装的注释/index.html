比较转录组分析（四）—— 组装的 GO 注释

发表于2022-11-28|更新于2022-11-28|学习生信
|字数总计:3.8k|阅读时长:12分钟|阅读量:

前言

之前我们已经讨论了评估组装质量和组装去冗余的方法，在这里我们再来进一步谈一谈组装的注释。

这一环节应该算得上是转录组的第一个重头分析部分，因为它可以让执行者对自己的组装有一个更清晰的认知。

什么是注释

注释分很多种，但是它们在转录组分析中的最终目的都指向一个共同点 —— 解释组装。

比如说 GO 注释、KEGG 注释、Pfam 注释、nr 注释这些，它们的注释内容是不一样甚至可以说五花八门的，但是无非都是想说明组装中不同的 contig 它代表的是哪一种基因或者说它执行了什么样的生物学功能。从整体分析流程的角度出发，通过注释，我们可以了解在差异表达分析中展现出显著上调或者显著下调的基因它到底在生物体里是干什么的。比方说，一个显著下调的基因通过注释我们知道了它编码热休克蛋白（又或者是负责和应激反应有关的功能），我们就可以说这个基因在处理组中对应的 mRNA 丰度比对照组显著更低，可能反映了什么现象，再进一步推断出结论。

此外我们也可以通过注释的情况观察组装的转录组在不同功能上的分布，或者关注某一类特定基因的表达情况等。

不过首先我们得知晓，所有的注释库都是人为编写的，并且对于每个基因的注释也都是人为划定的，关于这一点有很多可深入了解的地方，关于 GO 注释可以看一看这两篇文章（都概括的非常不错）：GO注释和富集分析概念介绍 from 简书 与 Gene Ontology(GO)简介与使用介绍 from 知乎。

上述内容也仅作了解之用，有个概念性的了解（知道自己后续的分析是在干什么）就没问题了。

下文我将就 GO 注释为主进行分享，其他的注释如果后续有机会也可能会单独出文介绍介绍（挖坑）。

GO 注释

因为前文提到的文章里已经将重要的概念都差不多介绍了一下，这里我就挑取其中重要的几点再详细说说并补充几点可能需要注意的事情。

①、GO 注释被分为三大类，分别是生物过程、细胞组分和分子功能。

②、GO terms 之间可能是有关系的，比如一个 term 是另外一个 term 的父级项，整个 GO 的结构是一个有向无环图。

关于第 ② 点，我随便举个例子（不是 GO term 的），比如说 吃饭 和 吃下午饭 这两个 term 里，明显可以看出 吃饭 是 吃下午饭 的父项，因此在某个基因上如果出现了 吃下午饭 这个 term，那么它就同时拥有 吃饭 这个 term。

③、基于上述原因，一个基因中可能会出现非常多相关的 GO term 注释，而在后续的富集分析中也会表现出来（比如说前几的都是一些相关联的 GO term），因此在后续的富集分析里可以考虑把一些相关的 term 给删掉一部分，仅保留具有代表性的那个。

④、GO term 的注释事实上是相对宽松的，只要这个基因跟某个功能沾上了边就会有人给它这个功能对应的注释。

⑤、GO term 是一个一直在保持更新的注释库，因此老旧的注释库数据可能存在某个 term 被废弃掉的情况，因此在某个项目的分析过程中最好保持数据库版本的一致，并在不需要继续使用的时候抽空更新数据库（同时记录好使用过的版本、做好版本控制，保证结果的可再现性）。

方法

就 GO annotation 而言，可以选择的软件有很多，比如 Interproscan、eggnog-mapper、Trinotate、Blast2GO 等等，包括一些在线网站比如像 Pannzer2 等也能够完成这一任务。

在我所介绍的以上软件里我用过的只有 Interproscan 和 Trinotate，Blast2GO 由于是商业化软件不好弄所以没有条件的友们可以直接放弃这一个选项。Interproscan 和 Trinotate 中我使用的是后者，因为我老板一直用的也都是这个。

至于哪个注释软件最准确，因为我没有横向比较过它们之间的优缺点所以我也并不确定，总的来说通过实践我觉得 Trinotate 是没问题的，Interproscan 相比之下配置较为简单但也非常要求计算资源。

最值得注意的应该是 Trinotate 的开发者也开发过 Trinity ，两个软件之间的衔接非常好，甚至有一些 Trinotate 的功能就是专门为 Trinity 的组装提供的，所以后续我会就 Trinotate 的注释方法进行分享介绍。

Trinotate 安装

这一点可以参照 Trinotate 官网进行（传送门），不过在这里提出几点可能可以帮助减少软件配置时间的建议：

①、如果仅使用 Trinotate 进行 GO 注释而不进行别的注释，那么仅安装 Software Required 中的 Trinotate、TransDecoder、SQLite 和 NCBI BLAST+ 就可以满足运行需求。

②、NCBI 的 blast 速度事实上挺慢的，如果转录组较大而且希望运行时间短一些，可以考虑使用 diamond blast 进行后续的步骤。但是有研究说 diamond blast 不如 NCBI blast 精准（但没办法它实在是太快了）。

③、可选项的用处不是很大，如果想要拓展功能再配置个 HMMER 就可以了（用于 Pfam 注释）。

后记

本来以为注释这一篇章可能会写的有些累，但没想到写的还挺快的，篇幅相较于前两篇而言也算是缩水不少，但事实上如果要从配置软件开始的话可能麻烦还是不少的，因为 Trinotate 的依赖项可丝毫不比前几个少。

事实上 GO annotation 这一步并不算难，但对于刚接触这一事物的生信苦手们可能会有很多令人感到难以解惑的点，而且网络上关于 GO 注释的教程确实也不算完善，也缺少一个统一的流程，我一开始的时候只在 bing 和百度上找这一步的教程花了我非常多的精力但也没有解决问题，我一开始甚至是直接打开 trinotate_annotation_report.xls 这个文件对 GO term 进行了分列和删除来统计注释，不仅笨还浪费时间浪费精力。

可能这也是我信守马克思主义中否定之否定规律的原因吧，找了那么久还真找出方法来了，也找出经验来了（指不懂的东西如果查不到就去 github 里找有没有人有相同的问题）。事物或许看上去是停滞不前的，但事实上它在螺旋上升、曲折前进。当然有人引着肯定是最好的事情，我看到过非常多篇精心制作的生信教程文章，里面不仅做教学还做科普，给我这个生信苦手带来了诸多福音。

这篇文在写的过程中可能主观性较强，里面或许落下了很多应该要讲的重点但未提及到，后续我还会不断检查如果有哪里需要补充就及时更新，虽然很大程度上是自娱自乐但是如果能帮到某位朋友也挺好。

题外话 —— 关于做 ppt 的技巧

感觉这个单独出也不太可能了，因为要讲的东西太多，现在写生信就已经够呛了估摸着这个不知道得拖到啥时候。

正好这一篇文篇幅不够，把这个拿来凑一凑长度也算不错。

首先几个正文的排版小技巧（都是从别的小组汇报里偷学的，而且大多数是我女朋友偷偷告诉我的）：

①、正文字体不是越大越好，12-16 看久了也挺好看的。不够关于这点也并不是固定不变，得看做的内容是什么，有些字体要放大些整体画面才会更有美感（但尽量一个汇报中的所有字体大小都统一一些）。

②、正文字体不宜过挤，1.4-1.7 的行距可以有效改善这个问题，最好再加个段前段后 3 磅。

其次几个需要注意的点：

①、一面 ppt 的字不要太多，最好做到图文结合。要讲的东西不一定要全放在 ppt 上，但关键的内容一定不能缺。可以用几个短句把要讲的点精准概括下。

②、如果是学术报告，就不要塞太多动画！不要塞太多动画！不要塞太多动画！（也不是不能就是别弄花里胡哨）

最后几个小技巧：

①、活用加粗、项目符号与编号可以让层次感更多样，不易审美疲劳。

②、右下角最好有页脚编号，能让别人提问时清晰地指出问题在 ppt 的第几页。

③、WPS 自带的各种图标图形也可以很好地丰富画面。

④、想做出自己觉得好看的 ppt，最有效的方式就是去看自己喜欢的其他 ppt 是怎么做的。

加分项：

①、网上的模板可以套用，但比较难满足自己的所有需求，所以可以自己做个 ppt 模板或者是照葫芦画瓢把网上的模板修改成自己的。

（比如我有一个模板就是照着其他小组大佬的风格弄的，不过也有个是原创 :D）

②、关于各个元素的布局事实上也很有考究，有兴趣可以学习一些绘画中的构成知识。
