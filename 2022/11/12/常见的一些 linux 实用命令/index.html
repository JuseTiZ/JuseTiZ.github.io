前言

这次文涉及到的主要是我在使用 linux 系统时会常使用的命令。

这些命令的灵活使用，会让在 linux 系统上的操作更加可靠。

这个栏目可能目前并不完善（有些可能我一时半会想不起来），之后我会不断检查并更新。

阅读之前可能需要知道的知识：

管道符、标准错误、标准输出等概念

线程、内存是什么以及它们之间的联系和区别

常用的命令大全

screen —— 运行长时命令的最优选

闲杂话

我曾有过这样的苦恼：

如果我要运行一个需要特别久的命令，我该怎么办？总不可能电脑一直挂着 linux 不退吧。

如果是在自己的电脑上运行，可能方法确实只有这一个了，毕竟电脑关了命令肯定也没法跑下去啊。

但是如果是在服务器上运行，那么 screen 这个命令，将会起到非常显著的作用。

执行

screen 这个命令干了什么？简略地说，它的作用就是在服务器上创造了一个如果你不手动终止那么将一直存在的终端，它不会随着你关闭自己的终端而消失，而是会一直存在于服务器的后台里，并执行其中运行的命令。

题外话

能起到相同作用的命令还有 nohup 这个已在其他文章中提到的指令。

但对于我而言 nohup 最大的作用还是记录屏幕输出，使用 screen 操作和运行长时命令还是更上选，此外 screen 还可以保留进入时的环境配置，因此不用担心在后续的变动会影响到里面命令的执行。

df -h & htop

这两个都是相当简单且直观的命令。

其中 df -h 可以让执行者知道还剩余多少空间，当要跑的命令会输出非常多非常大的文件时，应当提前使用这个命令判断会不会挤爆硬盘。（当空间被挤爆的时候命令会异常中断，而且这种中断是没法重新续上的）

htop 的作用就是直观地体现线程和内存的占用情况，可以根据剩余的内存合理地设置软件的相关参数。

不过需要注意的是，并不是用的线程和内存越多命令就跑的越快，不过整体而言运行花费的时间会和使用的内存成反比。

nohup

这个在上面已经提到过了，这里再提一遍，一是起到水字数的作用，二是起到强调的作用。

nohup 英文全称 no hang up（不挂起），用于在系统后台不挂断地运行命令，退出终端不会影响程序的运行。

那么除了这个作用以外，它还会输出一个名叫 nohup.out 的文件到当前目录下，这个文件记载了所有该指令本来会有的屏幕输出，可能乍一看意义不大，但是很多在写 paper 时会用到的信息都可能在里面，此外也可以通过查看这个文件了解到命令的执行情况，在运行出错后也可以快速地了解到具体的问题在哪。

对于所有在运行时会把有用信息在屏幕输出上而不保存在某个文件里的命令（例如 Bowtie2 Trimmomatic 等），都应当在前面加一个 nohup 以进行记录，nohup 不会覆盖已有的 nohup.out，而是会在最后进行 append。

此外它还可以让执行者退出终端仍不中断命令，这种只有益处没有害处的双重保险加上又何乐而不为呢？

将正在运行的命令放置在后台

关键命令：bg

应该算是一个很常规的技巧？

首先 ctrl+z 把命令放入后台，但此时命令会处于暂停状态。

此后输入 bg 可以使得它继续执行（如果要对特定命令可以先用 job 查看被中断的命令编号然后 bg %编号）。

如果该命令会有屏幕输出，那么在后续的工作中虽然你可以在终端执行各种命令但是还是会不时跳出之前命令的消息，所以运行前加个 nohup 真的挺不错。

此外，在命令的最后面加上 & 也能起到同样的作用。

中断命令

关键命令：kill

和上面一样是个也算很常见的命令。

使用 job 可以看到不同工作的编号，然后把要 cut 的命令用 kill %编号 杀死。

但一般像这种会把已有的工作给断掉的命令都要谨慎使用。

提高进程可以打开的最大文件数

关键命令：ulimit -n xxx

当运行需要操作非常多文件的命令时，它有时会报错说无法打开那么多的文件，那么就可以通过这个命令来解决。

ulimit -a 可以用来查看各项限制，其中 open file 项即我们要修改的。

当然在其他的服务器上（学院或老板的呀之类的），这个命令也是有上限的，超出特定值时会被禁止。

但要用到非常多文件的时候并不多，所以不用担心。

终焉的禁忌之术 —— sudo rm -rf /

业界俗称删库跑路。记住，在任何情况下都应当小心地使用 rm -rf ，linux 没有回收站，也没有备份。

所有的文件在删除前都需小心思考再思考！就算跑出来的文件是错的或者是未符合预期的，都应当好好地评判是否有很大的必要将其马上删除，在大多数时候，所剩的空间都并不会对于保存这些文件有所阻碍。当你发现或许还用的上这个文件但它已经被删除时，所带来的麻烦将比输入 rm -rf 大得多。
